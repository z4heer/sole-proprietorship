# System Design Fundamentals

## 1. System Design

### System Design kya hai

Large software systems ko plan karna.
Kaunsa component kya karega, kaise connect hoga, scale kaise hoga.

### Iska use kisliye hota hai

* Large applications build karne
* High traffic handle karne
* Reliable systems banane
* Interview discussions me

### Ye internally kaise kaam karta hai

Application ko multiple parts me divide kiya jata hai:

* Client
* API
* Service
* Database
* Cache
* Queue

Har layer ka specific role hota hai.

### Working Steps

User → Load Balancer → API Server → Service → Database → Response

### Mental Image / Analogy

Mall building planning jaisa:

* Entry gate
* Security
* Shops
* Billing
* Parking

Sab organized hona chahiye.

### Real Engineer Usage

* E-commerce systems
* Banking apps
* Food delivery
* OTT platforms

### Common Tools

* Draw.io
* Excalidraw
* Lucidchart
* Postman

### Troubleshooting Thinking

* Bottleneck kaha hai?
* Slow layer kaunsi hai?
* Single point failure hai kya?

### Communication Practice

* “System ko loosely coupled design karna important hai.”
* “Scalability aur fault tolerance ko start se consider karna chahiye.”
* “Main traffic flow aur database load ko separately analyze karta hu.”

## 2. Load Balancer

### Load Balancer kya hai

Traffic ko multiple servers me distribute karta hai.

### Iska use kisliye hota hai

* Server overload avoid karne
* High availability
* Better performance

### Ye internally kaise kaam karta hai

Incoming requests ko:

* Round Robin
* Least Connection
* Weighted routing

ke through distribute karta hai.

### Working Steps

User → Load Balancer → Server-1 / Server-2 / Server-3

### Mental Image

Traffic police vehicles ko different roads par bhej raha hai.

### Real Engineer Usage

* Nginx
* AWS ELB
* HAProxy

### Common Commands

```bash
systemctl status nginx
```

### Troubleshooting Thinking

* Ek server zyada load le raha?
* Health checks fail ho rahe?
* Sticky session issue?

### Communication Practice

* “Load balancer traffic distribution improve karta hai.”
* “Ye high availability architecture ka important part hai.”

# Monolith & Microservices

## 1. Monolith Architecture

### Monolith kya hai

Pure application ka single codebase aur deployment.

### Iska use kisliye hota hai

* Simple projects
* Fast initial development

### Ye internally kaise kaam karta hai

Frontend + Backend + Business logic ek hi application me.

### Working Steps

User → Single App → Database

### Mental Image

Ek bada shopping mall jisme sab kuch ek building me.

### Real Engineer Usage

* Small startups
* Legacy enterprise apps

### Troubleshooting Thinking

* Small change → full deployment
* Scaling difficult
* One bug → full app impact

### Communication Practice

* “Monolith simpler hota hai initially.”
* “Lekin scale hone par maintenance difficult ho sakta hai.”

---

## 2. Microservices

### Microservices kya hai

Application ko small independent services me divide karna.

### Iska use kisliye hota hai

* Independent scaling
* Faster deployments
* Better modularity

### Ye internally kaise kaam karta hai

Har service:

* Own logic
* Own deployment
* API communication

### Working Steps

### Mental Image

Different shops in a market.

### Real Engineer Usage

* Netflix
* Amazon
* Swiggy

### Common Tools

* Docker
* Kubernetes
* Kafka

### Troubleshooting Thinking

* Service-to-service latency
* Network failure
* API timeout

### Communication Practice

* “Microservices independently scalable hote hain.”
* “Service isolation fault impact reduce karta hai.”
