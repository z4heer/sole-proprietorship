# 🟢 SYSTEM DESIGN (SINGLE)

---

## 🧠 SIMPLE MEANING

System Design means:
👉 Planning how all parts of a system work together.

---

## 🎬 STEP-BY-STEP FLOW

Let’s see:

1. User sends request

2. Request goes to system

3. System processes
   → backend, database, cache

4. System handles load
   → scaling, queue

5. System returns response

👉 Flow:

**User → System → Process → Scale → Response**

---

## 🔩 MAIN PARTS

### 1. Components

* Backend, DB, Cache, API

---

### 2. Communication

* How parts talk

---

### 3. Scaling

* Handle more users

---

### 4. Reliability

* Handle failures

---

## 🔍 SIMPLE THINKING

### ⚡ How it becomes good

* Efficient flow
* Proper scaling

---

### ❌ What can go wrong

* Poor design → slow system
* No scaling → crash

---

### 🔄 How data moves

* User → API → Backend → DB → Response

---

### ⚙️ What is happening inside

* Multiple components interacting

---

> Okay… system design is planning system…
> how backend… database… cache work together…

# 🟢 CLIENT-SERVER MODEL (SINGLE)

---

## 🧠 SIMPLE MEANING

Client-Server Model means:
👉 One system (client) requests data, another system (server) provides it.

---

## 🎬 STEP-BY-STEP FLOW

1. Client sends request

2. Request goes through network

3. Server receives

4. Server processes

5. Server sends response

6. Client shows result

👉 Flow:

**Client → Network → Server → Response → Client**

---

## 🔩 MAIN PARTS

### 1. Client

* User side (browser/app)

---

### 2. Server

* Backend system

---

### 3. Network

* Connects both

---

## 🔍 SIMPLE THINKING

### ⚡ How it becomes fast

* Fast server
* Low latency

---

### ❌ What can go wrong

* Server down
* Network issue

---

### 🔄 How data moves

* Client → Server → Client

---

### ⚙️ What is happening inside

* Request/response cycle

---

## 🚶‍♂️ SPEAKING PRACTICE

Say this:

> Okay… client-server means…
> client sends request…
> server processes…
> and sends response back…
> so client asks… server answers…

---

## 🧠 MIND RULE

* Think: **request → process → response**
