# 📘 **1. Skill: Frontend vs Backend**

## 1. 🎯 One-Line Definition

Frontend handles user interface; backend handles logic and data processing.

## 2. 🧭 Purpose

* Separation of concerns
* Scalable architecture

## 3. 🧱 Core Concept

Frontend = UI
Backend = Brain

**Analogy:**
Restaurant → Menu/UI vs Kitchen/Processing

## 4. ⚙️ Internal Working

1. User interacts with UI
2. Frontend sends request
3. Backend processes
4. Response displayed

## 5. 🔄 System Flow

User → Frontend → Backend → DB → Response → UI

## 6. 🔗 Connected Systems

* APIs
* Database
* Client-server

## 7. 💻 Practical Execution

* Angular (frontend)
* Java/Python (backend)

## 8. ⚠️ Common Mistakes

* Mixing logic in frontend
* Tight coupling

## 9. 🧪 Debugging Mindset

If UI issue → check frontend
If data issue → check backend

## 10. 📌 Interview Explanation

“Frontend manages UI while backend handles business logic and data.”

## 11. 🧠 Memory Hooks

* Trigger: **UI vs Logic**
* Analogy: Menu vs Kitchen

## 12. 🔁 Active Recall

* What does frontend do?
* What does backend do?

# 📘 **2. Skill: REST API**

## 1. 🎯 One-Line Definition

REST API enables communication between client and server using HTTP methods.

## 2. 🧭 Purpose

* Standard way to interact with backend

## 3. 🧱 Core Concept

REST = **Resource-based communication**

**Analogy:**
Ordering items via menu options

## 4. ⚙️ Internal Working

1. Client sends HTTP request
2. Server processes resource
3. Returns response

## 5. 🔄 System Flow

Client → API Endpoint → Backend → DB → Response

## 6. 🔗 Connected Systems

* HTTP
* JSON
* Backend

## 7. 💻 Practical Execution

* GET /users
* POST /users

## 8. ⚠️ Common Mistakes

* Wrong HTTP methods
* Poor API design

## 9. 🧪 Debugging Mindset

If API fails → check:

* Endpoint
* Request/response

## 10. 📌 Interview Explanation

“REST API allows communication using HTTP methods for resource operations.”

## 11. 🧠 Memory Hooks

* Trigger: **GET, POST, PUT, DELETE**
* Analogy: Menu actions

## 12. 🔁 Active Recall

* What is REST?
* What are HTTP methods?

# 📘 **3. Skill: JSON Data Exchange**

## 1. 🎯 One-Line Definition

JSON is a lightweight format used for data exchange between systems.

## 2. 🧭 Purpose

* Standard data format for APIs

## 3. 🧱 Core Concept

JSON = **Key-Value data**

**Analogy:**
Dictionary (word → meaning)

## 4. ⚙️ Internal Working

1. Data structured in key-value
2. Sent in request/response
3. Parsed by systems

## 5. 🔄 System Flow

Frontend → JSON → Backend → JSON → Frontend

## 6. 🔗 Connected Systems

* REST API
* Frontend
* Backend

## 7. 💻 Practical Execution

```json
{
  "name": "John",
  "age": 30
}
```

## 8. ⚠️ Common Mistakes

* Invalid format
* Missing keys

## 9. 🧪 Debugging Mindset

If data issue → check:

* JSON structure
* Parsing errors

## 10. 📌 Interview Explanation

“JSON is a lightweight format used for structured data exchange in APIs.”

## 11. 🧠 Memory Hooks

* Trigger: **Key → Value**
* Analogy: Dictionary

## 12. 🔁 Active Recall

* What is JSON?
* Why used in APIs?

# 📘 **4. Skill: Authentication & Authorization**

## 1. 🎯 One-Line Definition

Authentication verifies identity; authorization controls access.

## 2. 🧭 Purpose

* Secure systems

## 3. 🧱 Core Concept

AuthN = Who are you
AuthZ = What can you do

**Analogy:**
Login + access permissions

## 4. ⚙️ Internal Working

1. User logs in
2. Identity verified
3. Permissions checked

## 5. 🔄 System Flow

User → Login → Token → Access control → Resource

## 6. 🔗 Connected Systems

* Security
* APIs
* Sessions

## 7. 💻 Practical Execution

* JWT tokens
* Session-based auth

## 8. ⚠️ Common Mistakes

* Confusing auth vs authorization
* Weak password handling

## 9. 🧪 Debugging Mindset

If access denied → check:

* Token
* Permissions

## 10. 📌 Interview Explanation

“Authentication verifies identity while authorization determines access rights.”

## 11. 🧠 Memory Hooks

* Trigger: **Who + What allowed**
* Analogy: Login + roles

## 12. 🔁 Active Recall

* Difference Auth vs Authorization?
* What is JWT?
